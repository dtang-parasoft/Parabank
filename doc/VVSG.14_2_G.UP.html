Avoid unused parameters [VVSG.14_2_G.UP]



DESCRIPTION 


"The variable's value is assigned but never used, making it a dead store."
[CWE-563]

This rule identifies method parameters that are never used.
SCOPE LEVEL 

LINE
SINCE 

v4.5
NOTES 

By default, the rule considers primitive type parameters that are assigned
with a value but are never read to be used and does not trigger when such 
parameters are detected. The rule can be parameterized to report primitive
type parameters that are assigned with a value but are never read
(see PARAMETERS).

Non-primitive type parameters that are assigned with a value are considered
to be used and are never reported.
SECURITY RELEVANCE 

"This weakness could be an indication of a bug in the program or a deprecated
variable that was not removed and is an indication of poor quality. This could
lead to further bugs and the introduction of weaknesses." [CWE-563]
PARAMETERS 

-Check "public" methods. (enabled by default)
-Check "protected" methods. (disabled by default)
-Check package-private methods. (enabled by default)
-Check "private" methods. (enabled by default)
-Check that the parameter is read for parameters with primitive types (disabled
 by default)
    -If this parameter is enabled, the rule will trigger when a primitive type
     parameter of is assigned with a value but is never read. 
-Check "empty" methods (disabled by default)
    -If this parameter is enabled, the rule will trigger if a method is empty,
     contains only a return statement, or contains only a throw statement.
BENEFITS 

This rule helps you remove unused method parameters from your code to
avoid confusion and improve code security (see SECURITY RELEVANCE).
DRAWBACKS 

The rule may report false positives for recursively called methods when
the argument is directly passed to that method invocation.

In the following example, the 'test' method is called on an object of
the 'ChildNode' class that extends the 'BaseNode' class. 

public class BaseNode {
    public void test(boolean bCompactView) { 		//VIOLATION
        ChildNode child = getChild();
        child.test(bCompactView); 
    } 
}
EXAMPLE 

In the following example, the 'userId' parameter of the 'addProduct' method
is never used.

public class UP {
    
    public void addProduct(int userId, Product product) { // VIOLATION the 'userId' parameter is never used
        addProduct(product);
    }
}
REPAIR 

You can fix the code by removing the unused parameter or modifying the code
so that the parameter is used.

public class UP {
    
    public void addProductFixed(int userId, Product product) throws UnsupportedUserOperationException {
        if (!hasUserPrivileges(userId)) { // FIXED the 'userId' parameter is used
            throw new UnsupportedUserOperationException();
        }
        addProduct(product);
    }
}
