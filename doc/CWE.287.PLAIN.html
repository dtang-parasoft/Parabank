Password information should not be included in properties file in plaintext [CWE.287.PLAIN]



DESCRIPTION 


This rules checks if properties files have critical data stored as property values.
Critical data such as password strings should be stored using hash functions.
If the value is not hash sums, then a violation is reported.
SINCE 

v9.2
NOTES 

This rule assumes that passwords are stored using cryptographic hash functions with:
 - 128 bits sums - example: MD5 and RIPEMD-128
 - 160 bits sums - example: SHA-1 and RIPEMD-160
 - 320 bits sums - example: RIPEMD-320
 - 224 bits sums - example: SHA-224 and HAVAL-224
 - 256 bits sums - example: SHA-256, HAVAL-256, PANAMA, GOST and RIPEMD-256
 - 384 bits sums - example: SHA-384
 - 512 bits sums - example: SHA-512, WHIRPOOL

A violation will be reported for using hash sums shorter than 128 bits or using encoded passwords
The rule assumes that hash sums are stored as hexadecimal values.
SECURITY RELEVANCE 

"Storing a password in plaintext may result in a system compromise." [CWE-256]
"[If] the application stores sensitive information in cleartext in a file, or
on disk. The sensitive information could be read by attackers with access to
the file, or with physical or administrator access to the raw disk." [CWE-313]
PARAMETERS 

-List of property keys to check (RegExp)
BENEFITS 

Password information should not be included in properties file as plain text.
When application keeps plain text passwords in a properties file, then anyone
who can read this file can also read the user password.
If passwords must be placed in properties files, they should be hashed.
DRAWBACKS 

This rule reports false positives when password are stored using different functions than those described
in the notes.
EXAMPLE 

pass=123
REPAIR 

//use cryptographic hash function
pass=1f3870be274f6c49b3e31a0c6728957f
