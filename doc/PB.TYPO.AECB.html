Avoid "try", "catch" and "finally" blocks with empty bodies [PB.TYPO.AECB]



DESCRIPTION 


This rule identifies "try", "catch" and/or "finally" blocks with empty bodies. 
An error is reported for each occurrence.
SCOPE LEVEL 
LINE
SINCE 

v6.0
NOTES 

This rule will not flag an error in the following cases:
For catch block, if the exception variable name matches the regular expression.
SECURITY RELEVANCE 

category: Unsafe Error Handling and Logging

Empty "catch" block indicates unhandled errors. It is important for the 
application to handle every error appropriately.
PARAMETERS 

-Check "try" block (default as true)
-Check "catch" block (default as true)
-Check "finally" block (default as true)
-Disallow blocks with comments as content (default as false)
    By default, blocks with comments in them are treated as non-empty.
    With option turned on, blocks with only comments in them will be flagged.

-Regular expression of exception variable name.
    If the exception variable name matches the listed regular expression,
    this rule will not flag an error.
    Default regular expression is "^ignore$".
BENEFITS 

Empty blocks do not contribute any semantic meaning to the code and should be
removed for simplicity. Besides, it is good practice to insert error handling code 
inside of each "catch" block. Finally block should also contains clean-up code to
correctly implement its intended functionality.
EXAMPLE 

package examples.rules.pb.typo;

public class AECB {
    /*
     * VIOLATIONS: empty catch and finally blocks
     */
    public void method () {
        try {
            System.in.read ();
        } catch (java.io.IOException e) {
        } finally { 
        }
    }
}
REPAIR 

Add exception handling code inside the "catch" block.

package examples.rules.pb.typo;

public class AECBFixed {
    public void method () {
        try {
            System.in.read ();
        } catch (java.io.IOException e) {
            Messages.log(e); //FIXED
        } //FIXED, empty "finally" block is removed
    }
}
