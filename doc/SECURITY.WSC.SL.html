Avoid string literals except in constant declarations and calls to System.out or System.err's 'print' or 'println' methods [SECURITY.WSC.SL]



DESCRIPTION 


Managing string literals that are scattered over the code is difficult and
error-prone. It is a good practice to group them as string constants or store
them in a separate file to facilitate the maintenance of output messages.

This rule detects when string literals are used in calls to the following
methods:
- System.err.print()
- System.err.println()
- System.out.print()
- System.out.println()
SINCE 

v6.0
NOTES 
  
The rule does not report sting literals that are used in string constant
declaration statements and classes that extend the javax.servlet.http.HttpServlet
class.
SECURITY RELEVANCE 

Hard-coded constant strings make code hard to maintain and error-prone.
This brings a risk of introducing security vulnerabilities.
BENEFITS 

This rule helps you improve the maintainability of string literals in your
codebase.
EXAMPLE 

The following example shows a hard-coded sting literal "Hello World".

package examples.rules.security;
public class SL {
public static void main(String args[]) {
String msg= "Hello World!"; //VIOLATION
System.out.println(msg);
}
}
REPAIR 

You can fix the code by declaring a string constant that is later used
to print out the message.

package examples.rules.security;
public class SLFixed {
static final String WELCOME_MESSAGE= "Hello World!";
public static void main(String args[]) {
String msg= WELCOME_MESSAGE; //FIXED
System.out.println(msg);
}
}
