Threads & Synchronization [TRS] 

Rules related to threads & synchronization.
RULES

Do not use Atomic variables when always accessed in synchronized manner [TRS.AIL]
Do not use 'notify()'; use 'notifyAll()' instead so that all waiting threads will be notified [TRS.ANF]
Avoid accidental use of "Thread.interrupted()" [TRS.ATI]
Do not use variables of the unsafe type 'java.lang.ThreadGroup' [TRS.AUTG]
Do not use 'Thread.yield()' because it may behave differently under different Virtual Machines [TRS.AUTY]
Avoid invoking methods using "static" 'Calendar', 'DateFormat', or 'SimpleDateFormat' [TRS.CDF]
Use ConcurrentHashMap instead of Hashtable and "synchronizedMap" wrapped HashMap when possible [TRS.CHM]
Do not catch InterruptedException except in classes extending Thread [TRS.CIET]
Use ConcurrentLinkedQueue instead of Vector and synchronizedList when possible [TRS.CLQ]
Avoid compound synchronized collection accesses which violate atomicity [TRS.CMA]
Do not cause deadlocks by calling a "synchronized" method from a "synchronized" method [TRS.CSFS]
Do not call the "start" method of threads from inside a constructor [TRS.CSTART]
Do not let "this" reference escape during construction [TRS.CTRE]
Avoid unsafe implementations of the "double-checked locking" pattern [TRS.DCL]
Do not use DiscardOldestPolicy with PriorityBlockingQueue [TRS.DOPQ]
Do not use "getState" except for debugging purposes [TRS.GSD]
Inspect accesses to "static" fields which may require synchronization [TRS.IASF]
Make lazy initializations thread-safe [TRS.ILI]
Do not catch 'IllegalMonitorStateException' since this exception likely indicates a design flaw [TRS.IMSE]
Implement Runnable instead of extending Thread [TRS.IRET]
Do not call the 'run()' method directly on classes extending 'java.lang.Thread' or implementing 'java.lang.Runnable' [TRS.IRUN]
Do not call the 'start()' method directly on Thread class instances [TRS.ISTART]
Ensure that nested locks are ordered correctly [TRS.LORD]
Access related Atomic variables in a synchronized block [TRS.MRAV]
Give subclasses of Thread a 'run()' method so they can run as separate threads [TRS.MRUN]
Ensure threads are named [TRS.NAME]
Use "synchronized" blocks instead of making the whole method declaration "synchronized" [TRS.NSM]
Ensure 'wait()', 'notify()' and 'notifyAll()' are invoked on an object that is clearly synchronized in its enclosing mod scope [TRS.NSYN]
Avoid overriding synchronized methods with non-synchronized methods [TRS.OSNS]
Release Locks in a "finally" block [TRS.RLF]
Use synchronization on methods that implement 'Runnable.run()' [TRS.RUN]
Do not synchronize on constant Strings [TRS.SCS]
Do not synchronize on the class object returned by the 'getClass' method [TRS.SGC]
Do not mix "static" and non-"static" "synchronized" methods [TRS.SNSM]
Do not synchronize on a collection view if the backing collection is accessible [TRS.SOBC]
Do not perform synchronization using the "synchronized" keyword on implementations of "java.util.concurrent.locks.Lock" [TRS.SOL]
Do not synchronize on "public" fields since doing so may cause deadlocks [TRS.SOPF]
Do not synchronize on non-"final" fields since doing so makes it difficult to guarantee mutual exclusion [TRS.SOUF]
Manually synchronize on 'synchronized' collections when iterating over them [TRS.SSCI]
Synchronize access to "static" SimpleDataFormats [TRS.SSDF]
Make the get method for a field synchronized if the set method is synchronized [TRS.SSUG]
Do not perform synchronization nor call semaphore methods on an Object's 'this' reference [TRS.STR]
Avoid calling unsafe deprecated methods of 'Thread' and 'Runtime' [TRS.THRD]
Do not call 'Thread.sleep()' while holding a lock since doing so can cause poor performance and deadlocks [TRS.TSHL]
Avoid unsynchronized accesses of "Collections.synchronized" wrapped Collections [TRS.UACS]
Use unsynchronized Collections/Maps only when safe [TRS.UCM]
Use the same locking object to access variables [TRS.USL]
Do not start a thread without specifying a 'run()' method [TRS.UT]
Call 'wait()' and 'await()' only inside a loop that tests the liveness condition [TRS.UWIL]
Use 'wait()' and 'notifyAll()' instead of polling loops [TRS.UWNA]
Use the correct method calls on "java.util.concurrent.locks.Condition" objects [TRS.WOC]
Do not make the "writeObject()" method synchronized if no other method is synchronized [TRS.WOS]
