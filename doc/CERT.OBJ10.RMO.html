Avoid referencing mutable fields [CERT.OBJ10.RMO]



DESCRIPTION 


This rule identifies "static" mutable fields which are assigned into variables.
An error is reported for each occurrence.
SINCE 

v8.4
SECURITY RELEVANCE 

This rule helps to enforce the PCI DSS (Payment Card Industry Data Security 
Standard) Requirement #6: "Develop and maintain secure systems and 
applications".  Specifically, this rule helps to test for Issue 6.5.4: "Insecure 
direct object references".

Enforcing this rule will help to protect against the OWASP 2013 Top 10 
application vulnerability "A4-Insecure Direct Object References".
PARAMETERS 

-Check "static final" fields only (default false)
-Check fields of accessibility
    -public (default true)
    -protected
    -package-private
    -private
-Ignore fields of the following classes: (Glob expression)
    -default list
       * java.lang.System
       * java.util.Locale
BENEFITS 

If a "static" mutable field is assigned into a variable, then programmers have 
to be aware that any change to that variable will affect the state of the 
"static" mutable field directly.
EXAMPLE 

package examples.rules.pb.cub;

class NumberConst {

    public static final String[] NUMBER = {"1", "2"};
    
}

public class RMO {    
    public void run() {
        String[] number = NumberConst.NUMBER;  // VIOLATION
     
        // ......   
        number[0] = "XXX"; // New number
    }
}
REPAIR 

The variable 'number' should be assigned by a defensive copy of the mutable 
field instead of by the mutable field itself.

package examples.rules.pb.cub;

class NumberConst {

    private static final String[] NUMBER = {"1", "2"};
    
    public static final String[] getNumber() {
        String[] copy = new String[NUMBER];
        System.arraycopy(NUMBER, 0, copy, 0, NUMBER.length);
        return copy;
    }
}

public class RMOFixed {    
    public void run() {
        String[] number = NumberConst.getNumber();  // FIXED
        
        // ......   
        number[0] = "XXX"; // New number
    }
}
