Follow a naming convention for "boolean" getter methods [NAMING.GETB]



DESCRIPTION 


This rule identifies "boolean" getter methods whose names do not follow the 
user-defined naming convention (as specified in a regular expression).  By 
default, the rule requires that the name of each getter method begin with 'is', 
'can', 'has', 'have', 'are', 'was', or 'contains'.  An error is reported for 
each occurrence.
SCOPE LEVEL 
METHOD
SINCE 

v4.5
PARAMETERS 

-regular expression
    -"Boolean" getter method names should match the listed regular expression.  
    -The default is "^(is|can|has|have|are|was|contains).".  This enforces that 
     the names of "boolean" getter methods begin with "is", "can", "has", 
     "have", "are", "was", or "contains".
-check public method. (default is true)
-check protected method. (default is true)
-check package-private method. (default is true)
-check private method. (default is false)
-Negate regular expression
    -If this parameter is enabled, a violation will be flagged if the name of a
     "boolean" getter method matches the specified regular expression.
    -By default, this parameter is off.  This means that a violation will be 
     flagged for "boolean" getter method names which do not match the regular 
     expression specified.
-Ignore case
    -If this parameter is enabled, the regular expression will not be case
     sensitive.
    -By default, this parameter is off.  This means that the names of "boolean" 
     getter methods must match the regular expression specified, including in 
     case.
EXAMPLE 

package examples.rules.naming;

public class GETB {
    public boolean method () {  // VIOLATION
        return _ready;
    }
    private boolean _ready = false;
}
REPAIR 

Change getter method's name.

package examples.rules.naming;

public class GETBFixed {
    public boolean isReady () {  // FIXED
        return _ready;
    }
    private boolean _ready = false;
}
